RideShare Connect AI Handoff

High-level technical and product context for any engineer or AI system continuing work on this repository.

1. What This Project Is

RideShare Connect is a mobile-first smart carpooling platform for India. Its central idea is to match riders with partially overlapping routes, not only identical destinations. The repo now contains a real MVP foundation: Expo mobile app, Express backend, PostgreSQL schema, seed data, and a connected rider booking flow.

Passenger value

  • Partial-route ride matching
  • Lower shared fares
  • Optional mid-trip pickup for extra savings

Driver value

  • Better shared-ride economics
  • Foundation for return-trip monetization
  • Lower commission model

4. Current Rider Flow

  1. User passes through a demo phone / OTP login screen.
  2. User enters pickup, dropoff, and ride type.
  3. App calls previewRideMatches().
  4. Backend returns corridor-compatible matches.
  5. User chooses a match and then a vehicle option.
  6. App requests a quote from /api/bookings/quote.
  7. User books via /api/bookings.
  8. App shows an active trip screen with simulated trip progress.

5. Matching Logic

The current matching engine is corridor-based MVP logic. It does not yet use a real routing service. The backend:

  • infers a corridor from text pickup/dropoff
  • loads active trips from PostgreSQL
  • computes route overlap in kilometers
  • penalizes detour time
  • scores candidates and filters weak overlaps
  • returns derived vehicle and fare options
Seeded corridors: delhi_cp_noida and gurgaon_cp_central

10. Important Caveats

  • The login screen is demo-only. It does not send a real OTP and should not be mistaken for production auth.
  • Expo Go testing was blocked by local network reachability and unstable ngrok tunnel sessions. That is a transport problem, not a core app architecture problem.
  • The current web view is mainly a debugging surface and not a polished product target.
